# ТЗ — Игровые серверы CS2 для Esports.kz

> Своя инфраструктура серверов CS2 с минимальным пингом для игроков из Казахстана/СНГ. Используется для проведения матчей турниров платформы.

**Статус:** черновик от 2026-05-01.

---

## 1. Цели

- **Низкий пинг** (≤ 30 мс) для игроков КЗ.
- **Контроль качества**: свой античит, конфиги, демо-записи, логи.
- **Интеграция с сайтом**: сайт автоматически создаёт лобби для турнирных матчей, парсит результаты, забирает .dem-файлы для статистики.
- **Масштабирование**: 1 турнир = 8 матчей одновременно = до 16 серверов одновременно.

---

## 2. Можно ли несколько серверов CS2 на одной машине?

**Да, можно.** На одном dedicated server / VPS легко поднимаются 4-8 параллельных CS2-серверов через SteamCMD. Ограничения:

| Ресурс | На один CS2-сервер | На 8 серверов параллельно |
|---|---|---|
| **CPU** | 1 поток с пиками до 30% (одно ядро 3.5+ ГГц) | 8 потоков, **минимум 8-12 cores 3.5+ ГГц** |
| **RAM** | 1.5–2 ГБ при 10 игроках | **16–32 ГБ DDR4/DDR5** |
| **Сеть** | ~50 Mbps stable, low jitter | **1 Gbps unmetered/большой объём** |
| **Диск** | 50 ГБ под игру + место под demos | **500 ГБ NVMe SSD** (быстрая запись demo) |
| **Порты** | UDP 27015 (game) + 27020 (HLTV) + 27005 (client) | разные порты на каждый инстанс: 27015, 27016, ... |

**Критично:** настоящие CPU-cores, не виртуальные. NUMA-aware (если многосокетный сервер). Анти-DDoS на уровне провайдера.

## 4. Где размещать (геолокация для KZ-аудитории)

**Самый низкий пинг для игроков КЗ:**

| Регион | Хостер | Пинг для Алматы/Астаны | Цена |
|---|---|---|---|
| 🇰🇿 **Алматы / Астана** | KazTeleCom Cloud, Ducat, ICloud-Kazakhstan | **3–10 мс** | дороже, но идеал |
| 🇷🇺 Москва | Selectel, Timeweb, Servers.ru, Hetzner-RU | 60–80 мс | средне |
| 🇫🇮 Хельсинки | Hetzner, UpCloud | 90–110 мс | дешёво |
| 🇩🇪 Франкфурт | Hetzner, OVH | 130–150 мс | очень дёшево |

**Рекомендация:** на старте — 1 dedicated server в **Алматы или Астане**. Если бюджет позволяет, добавить второй в Москве как backup для российских игроков.

⚠ **Важно проверить у хостера:** разрешён ли gaming, есть ли реальный DDoS-щит (не просто IP-based filter), latency до целевых регионов через `mtr`/`traceroute` до выбора.

## 5. Software stack

### Базовый слой

```
Ubuntu 22.04 LTS Server
├── steamcmd (для установки игры)
├── linuxgsm (управление инстансами серверов)
├── docker / podman (опционально — для изоляции)
├── ufw firewall (открыть UDP 27015-27050)
└── fail2ban (защита от brute-force SSH)
```

### CS2 Server Software

- **steamcmd** — официальный установщик: `steamcmd +login anonymous +app_update 730 validate +quit`
- **CS2 Dedicated Server** (App ID 730) — стабильный из Steam
- **linuxgsm** или **csgsl** — обёртка для управления (старт, рестарт, обновление, мониторинг)
- **SourceMod + MetaMod** — плагины (хотя для CS2 поддержка ограниченнее чем CS:GO)
- **CS2KZ-server** или подобное — если нужны кастомные режимы

### Анти-чит

| Решение | Плюсы | Минусы |
|---|---|---|
| **VAC** (стандарт Valve) | Бесплатно, встроено | Лояльно к новым читам |
| **CS2 Match Anti-Cheat** | Включается опцией `mp_match_anti_cheat 1` | Только для match-серверов |
| **SMAC / SourceMod-плагины** | Кастомный | Поддержка для CS2 ограничена |
| **Своё решение через GSI + ML** | Гибко | Долго и сложно |

**Рекомендация для MVP:** `mp_match_anti_cheat 1` + ручная модерация admin'ом + обязательная запись demo для post-game review.

### Мониторинг

- **Prometheus** — метрики серверов (CPU, RAM, players online, ping)
- **Grafana** — дашборды
- **node_exporter** — system metrics
- **Custom exporter** для CS2 server stats через RCON / GSI

## 6. Конфигурация инстансов

Каждому CS2-серверу нужен свой порт:

| Инстанс | Game port (UDP) | HLTV port (UDP) | Client port |
|---|---|---|---|
| Server #1 | 27015 | 27020 | 27005 |
| Server #2 | 27016 | 27021 | 27006 |
| Server #3 | 27017 | 27022 | 27007 |
| ... | ... | ... | ... |

Запуск через linuxgsm:

```bash
# Для каждого инстанса — отдельная директория и lgsm-скрипт
./csgsserver-1 start
./csgsserver-2 start
# ... до 8 параллельных
```

В конфиге `server.cfg` для каждого:

```cfg
hostname "Esports.kz | Match #1234"
sv_password "матч-токен"  // только участники турнира
mp_match_can_clinch 1
mp_match_end_changelevel 1
tv_enable 1               // запись demo / hltv-трансляция
tv_autorecord 1
sv_logfile 1              // подробные логи
log on
```

## 7. Интеграция с сайтом Esports.kz

Это самое интересное — автоматизация турниров.

### Поток создания матча

```
Админ создаёт матч на сайте
   ↓
Сайт через RCON/SSH дёргает linuxgsm: start-server-with-config
   ↓
Сайт получает: IP, порт, пароль матча
   ↓
Сайт отправляет ссылку капитанам команд
   ↓
Команды заходят на сервер по ссылке steam://connect/IP:PORT/PASSWORD
   ↓
Игрок жмёт ".ready" в чате — сервер начинает
   ↓
По окончании сервер скачивает demo на хранилище (Supabase Storage / S3)
   ↓
Сервер парсит scoreboard, отправляет на сайт через webhook
   ↓
Сайт обновляет MatchPlayerStat и продвигает победителя по сетке
```

### Технические детали

- **RCON** — стандартный протокол управления Source-серверами. Используем `node-rcon` или `python-valve` в нашем backend.
- **Webhook на завершение матча** — пишем плагин SourceMod / Metamod, который при `mp_match_end` POST'ит результат на `https://esports.kz/api/internal/match-result`.
- **Demo upload** — после матча scp/rclone заливает .dem файл в Supabase Storage. Хранится 90 дней, потом архивируется.
- **GSI (Game State Integration)** — для live-обновлений на сайте (live scoreboard) сервер каждые ~100 мс шлёт JSON на наш endpoint.

### Безопасность

- RCON-пароль уникальный для каждого сервера, ротация раз в неделю.
- Webhooks подписываются HMAC SHA-256 + shared secret.
- Demo-файлы только для админов и игроков матча (signed URLs из Supabase).

## 8. Этапы внедрения

### Phase 1 — Pilot (1-2 недели)

- 1 VPS с 1 сервером CS2 (Алматы или Астана).
- Ручной запуск админом для конкретного матча.
- Demo вручную скачиваются и анализируются.
- Цель: провести 2-3 турнирных матча, отладить latency и стабильность.

### Phase 2 — MVP (1 месяц)

- 1 dedicated в Алматы с 4 параллельными серверами через linuxgsm.
- Автоматический запуск по кнопке в админке (RCON).
- Webhook на результат матча.
- Demo upload в Supabase Storage.

### Phase 3 — Production (2-3 месяца)

- Второй dedicated (Москва) для российских игроков.
- 8-16 серверов одновременно.
- Live-trans через GSI.
- Мониторинг через Prometheus + alerting.
- Auto-scaling: при росте нагрузки — автоматически поднять дополнительный VPS через API хостера.

### Phase 4 — Расширение (полгода+)

- Свои стрим-серверы (TV-сервера CS2, HLTV-релей).
- Match-making вне турниров (открытое лобби для подписчиков).
- Региональные серверы: Шымкент, возможно Ташкент.
