# ТЗ — Архитектура команд и составов

> Как организовать связь "Команда → Игроки → Игры" с учётом разных составов под разные дисциплины.

**Статус:** черновик от 2026-05-01.

---

## 1. Текущая модель (Team-per-game)

```
Team (id, name, tag, game, captainId, ...)
TeamMember (teamId, userId, role)
```

Каждая команда — отдельная сущность под одну игру.

**Ограничение:** игрок может быть в одной команде на каждую игру (контролируется в `joinTeam` action).

**Минус:** "Tulpar" в CS2 и "Tulpar" в Dota 2 — это две независимые сущности. Лого, описание, регион дублируются. История общая не видна.

---

## 2. Размеры составов

Закрепляем константой:

| Игра | Основной состав | Запасные | Всего |
|---|---|---|---|
| CS2 | 5 | до 2 | 5-7 |
| Dota 2 | 5 | до 2 | 5-7 |
| PUBG | 4 | 1 | 4-5 |

Реализация:

```ts
export const ROSTER_SIZE: Record<Game, number> = {
  CS2: 5,
  DOTA2: 5,
  PUBG: 4,
};
export const MAX_SUBS = 2;
export const MAX_ROSTER = (game: Game) => ROSTER_SIZE[game] + MAX_SUBS;
```

В UI показываем `members / ROSTER_SIZE[game]`, выходящих сверх — помечаем как `SUBSTITUTE`.

## 3. Три варианта эволюции модели

### Вариант A: оставить как есть (Team-per-game)

```
Team (id, name, tag, game, captainId, region, logoUrl, ...)
TeamMember (teamId, userId, role)
```

✅ Никаких миграций.
❌ Нельзя представить организацию с несколькими rosters.

**Когда выбрать:** проект только-только стартует, у команд один состав одной игры.

### Вариант B: Organization + Team (NaVi/Vitality-style)

```
Organization (id, name, tag, logoUrl, region, description, ownerId)
Team (id, organizationId?, name, game, captainId)  // organizationId опционален
TeamMember (teamId, userId, role)
```

- **Organization** — это бренд (NaVi, Vitality, FaZe, Cloud9, Astralis...)
- **Team** — конкретный roster одной дисциплины. Может принадлежать Organization или быть standalone (любительская команда).

✅ Правильная модель для серьёзных команд.
✅ Можно показать страницу организации со всеми её командами + общей статистикой.
✅ Standalone команды (без Organization) тоже поддерживаются.
❌ Больше моделей и UI: создание Organization → потом Team внутри.

**Когда выбрать:** есть реальные организации с несколькими rosters в нашей экосистеме.

### Вариант C: Team как бренд + Squad

```
Team (id, name, tag, region, logoUrl, description, captainId, privacy)
Squad (id, teamId, game, captainId, rating, maxRoster)
SquadMember (squadId, userId, role, joinedAt)
```

- **Team** — это и есть «бренд» (как Organization в варианте B). Без привязки к игре.
- **Squad** — состав под конкретную игру. У одной Team может быть до 3 Squad'ов (по одному на CS2/Dota/PUBG).
- Игроки состоят в Squad, не в Team напрямую.

✅ Один бренд с несколькими составами.
✅ Меньше миграций чем B (нет отдельной Organization-сущности).
✅ Простая ментальная модель: «Команда + Составы»
❌ Миграция текущих Team в новую структуру.
❌ Нужно консолидировать команды с одинаковым name (Tulpar CS2 + Tulpar Dota → один Team с двумя Squad).

**Когда выбрать:** готовы к рефакторингу, хотим бренд-уровень без полной Organization-модели.

## 4. Сравнительная таблица

| Критерий | A | B | C |
|---|---|---|---|
| Готовая миграция БД | ✅ ничего не нужно | ❌ Organization новая | ❌ Squad новая |
| Стоимость для пользователя | 0 | средне (новая модель) | средне (рефакторинг существующих команд) |
| Гибкость | низкая | максимальная | хорошая |
| Понятность | простая | средняя | средняя |
| Стоимость разработки | 10 мин | 2-3 часа | 1.5-2 часа |
| Когда подходит | сейчас, MVP | для зрелого проекта | компромисс |

## 5. Рекомендация

**Поэтапно:**

### Phase 1 (сейчас, 10 минут): остаёмся на A + хардкод rosterSize
- Добавить константу `ROSTER_SIZE` в `src/lib/games.ts`
- Использовать в проверках `joinTeam`, `approveJoinRequest`
- В UI показывать `5/5` или `4/4` корректно
- Если пользователь хочет 6-го игрока — он автоматически становится SUBSTITUTE

### Phase 2 (когда у нас 50+ команд и появятся реальные организации с 2+ rosters): миграция на C

**Алгоритм миграции:**
```sql
-- 1. Создаём новую таблицу Team_v2 без поля game
-- 2. Создаём Squad с FK на Team_v2 + game
-- 3. Группируем существующие Team по name (или name+region):
--    INSERT INTO Team_v2 (name, tag, region, logoUrl, ...)
--      SELECT name, tag, region, logoUrl FROM Team GROUP BY name
-- 4. Для каждого Team создаём Squad по его game
-- 5. Переносим TeamMember в SquadMember с FK на новый Squad
-- 6. Дропаем старую Team
```

Для пользователей это будет автоматический мерж: их CS2-команда и Dota-команда с одинаковым именем станут одной Team с двумя Squad.

### Phase 3 (опционально, для крупных орг): добавляем Organization
- Если 2+ команды хотят объединиться в брендовый юр. зонтик (NaVi-style) — добавляем Organization как родителя для Team.

---

## 6. Что в /admin будет нужно после Phase 2

**Кабинет суперадмина:**
- `/admin/teams` — список всех Team с количеством Squad'ов
- `/admin/teams/[id]` — управление: лого, имя, тег, описание + список Squad'ов
- `/admin/teams/[id]/squads/[squadId]` — управление составом одной игры
- `/admin/squads` — фильтрованный list по игре
- Импорт/экспорт ростеров

**Кабинет капитана:**
- `/teams/[tag]` — общая страница бренда
- `/teams/[tag]/cs2` — конкретный squad
- `/teams/[tag]/dota2` — другой squad
- В edit капитан управляет составом своего squad'а отдельно

---

## 7. Открытые вопросы

- [ ] Может ли один человек быть капитаном двух Squad'ов одного Team? → **Да, если он играет в обе игры.**
- [ ] Может ли один человек быть капитаном Team но не играть ни в одном Squad? → **Нет, капитан Team автоматически основной игрок одного из Squad'ов.** Или сделать Team-owner отдельной ролью без обязательной игры (тогда это уже Organization).
- [ ] Как хранить историческое расформирование составов? → Опциональный `disbandedAt` для Squad.
- [ ] Что делать с standalone-командой без бренда? → В варианте C — невозможно (Squad всегда внутри Team). В варианте B — Organization=null.

---

## 8. Final decision

**На текущий момент (MVP, 1-10 команд):** Variant A с константой ROSTER_SIZE — реализуем сейчас (10 минут).

**Через 1-3 месяца (50+ команд):** мигрируем на Variant C — единое имя бренда с несколькими составами.

**Variant B (Organization)** оставляем на случай, если придут крупные локальные организации с 5+ rosters в разных играх. Маловероятно для KZ-сцены в первый год.
